## 📚 整个调用链路（必须滚瓜烂熟！）

```
用户访问：GET http://localhost:8080/products/1?userId=5

    ↓ ① Tomcat 接收请求

    ↓ ② DispatcherServlet（前端控制器）
       问：这个 URL 应该由哪个 Controller 处理？
       答：@GetMapping("/products/{id}") → ProductController.getProduct()

    ↓ ③ ProductController.getProduct(id=1, userId=5)
       作用：接收 HTTP 请求，解析参数
       动作：调用 productService.getProductById(1)

    ↓ ④ ProductService.getProductById(id=1)
       【关键】这里有什么？
       - @Cacheable(cacheNames="product:detail", key="#id")
       
       第一次调用：
       1. Spring 代理拦截这个方法
       2. 去 Redis 查 key: "product:detail::1"
       3. 没命中 → 执行方法体 → productMapper.findById(1)
       4. 拿到结果 → 写入 Redis → 返回
       
       第二次调用：
       1. Spring 代理拦截这个方法
       2. 去 Redis 查 key: "product:detail::1"
       3. 命中 → 直接从 Redis 返回 → 不执行方法体

    ↓ ⑤ ProductMapper.findById(id=1)
       【关键】MyBatis 如何工作？
       1. MyBatis 扫描到@Mapper 注解
       2. 生成 JDK 动态代理
       3. 代理读取 ProductMapper.xml
       4. 找到 namespace="com.example.demo.mapper.ProductMapper"
       5. 找到 id="findById" 的 SQL
       6. 执行：SELECT * FROM product WHERE id = ?
       7. 参数绑定：PreparedStatement.setLong(1, 1)
       8. 执行查询，拿到 ResultSet
       9. 映射每一行到 Product 对象
       10. 返回 Product

    ↓ ⑥ MySQL 数据库
       真正执行 SQL，返回数据

    ← ⑦ Product 对象一路返回

    ← ⑧ ProductService 把 Product 写入 Redis 缓存

    ← ⑨ ProductController 收到 Product

    ← ⑩ Spring MVC 把 Product 转成 JSON
        {
          "id": 1,
          "name": "iPhone",
          "price": 899,
          "stock": 100
        }

    ← ⑪ 返回给浏览器
```

## 🎯 你必须搞懂的 3 个核心问题

### 问题 1：Spring 是如何"自动"工作的？

**关键词：扫描 + 代理**

1. **扫描**
   - `@SpringBootApplication` 扫描所有@Component/@Service/@Controller
   - `@Mapper` 扫描所有 Mapper 接口
   - 创建这些类的 Bean，放入 ApplicationContext

2. **代理**
   - `@Cacheable` → Spring 生成代理类，在方法前后加缓存逻辑
   - `@Transactional` → Spring 生成代理类，在方法前后加事务逻辑
   - `@Mapper` → MyBatis 生成代理类，执行 SQL

**面试标准答案：**
> "Spring 通过扫描注解发现组件，然后为带特殊注解（如@Cacheable、@Transactional）的类生成代理对象。代理对象会在原方法执行前后插入额外逻辑（如缓存、事务），这就是 AOP 的思想。"

---

### 问题 2：依赖注入（DI）是怎么回事？

**你现在的代码：**
```java
@RestController
public class ProductController {
    private final ProductService productService;
    
    // 构造函数注入
    public ProductController(ProductService productService) {
        this.productService = productService;
    }
}
```

**Spring 启动时：**
1. 扫描到 ProductController 有@RestController
2. 准备创建这个 Bean
3. 发现构造函数需要 ProductService
4. 从容器中取出 ProductService（之前已经创建了）
5. 调用构造函数，完成注入

**为什么不用 new？**
- ❌ `new ProductService()` → 你要手动 new ProductMapper，再 new SqlSessionFactory...
- ✅ Spring 帮你 new 好，你直接用

**面试标准答案：**
> "依赖注入是 Spring 的核心特性。传统方式需要手动创建依赖对象，而 Spring 会在启动时自动扫描所有 Bean，通过反射调用构造函数或 setter 方法，完成依赖注入。这样降低了耦合，方便单元测试。"

---

### 问题 3：缓存是如何生效的？

**你的代码：**
```java
@Cacheable(cacheNames = "product:detail", key = "#id")
public Product getProductById(Long id) {
    return productMapper.findById(id);
}
```

**底层发生了什么：**

**第一次调用：**
```
1. Spring 生成的代理对象拦截这个方法调用
2. 代理执行 CacheAspect 的逻辑
3. 从 Redis 查 key: "product:detail::1"
4. 没查到 → 执行原方法 → productMapper.findById(1) → 查 MySQL
5. 拿到 Product 对象
6. 用 GenericJackson2JsonRedisSerializer 序列化成 JSON
7. 写入 Redis: SET "product:detail::1" "{...}" EX 1800
8. 返回 Product 对象
```

**第二次调用：**
```
1. Spring 生成的代理对象拦截这个方法调用
2. 代理执行 CacheAspect 的逻辑
3. 从 Redis 查 key: "product:detail::1"
4. 命中缓存 → 反序列化 JSON 成 Product 对象
5. 直接返回，原方法不会执行
```

**面试标准答案：**
> "Spring Cache 基于 AOP 实现。启动时会扫描所有带@Cacheable 的方法，生成代理对象。当调用方法时，代理会先去缓存查，命中就直接返回，没有才执行原方法并写缓存。底层用的是 RedisTemplate 操作 Redis。"

### 进阶层（大部分人都知道）

5. **#{}和${}有什么区别？**
   - 预编译 vs 字符串替换
   - SQL 注入风险

6. **@CacheEvict 是在方法执行前还是后删除缓存？**
   - 默认在执行后
   - beforeInvocation=true 可以在执行前

7. **构造函数注入和@Autowired 注入有什么区别？**
   - final 修饰、避免 null、方便测试

---

### 高阶层（拉开差距）

8. **如果缓存 Redis 挂了，你的系统会怎样？**
   - 缓存穿透 → 所有请求打到 DB
   - 解决方案：本地缓存（Caffeine）+ 熔断降级

9. **如果两个用户同时修改同一个商品，缓存会怎样？**
   - 并发覆盖问题
   - 解决方案：分布式锁 / 先删缓存再更 DB

10. **你的@Cacheable 在什么情况下会失效？**
    - 同类中方法互相调用（代理失效）
    - 方法不是 public
    - 返回值是 null（但配置了不缓存 null）
